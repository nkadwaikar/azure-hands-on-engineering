# 🛡️ Azure Site Recovery (ASR)

A complete hands-on lab for configuring Azure Site Recovery, validating replication health, running test failover, and performing final failover.

> **Note:** Replication and failover behavior can vary by VM size, disk type, and region capacity. Validate all settings against your organization's resilience and governance requirements.

---

## 🎯 Learning Objectives

By the end of this lab, you will:
- Configure Azure Site Recovery for an Azure VM
- Understand source/target resources created during replication
- Validate replication health and RPO before failover
- Perform a test failover and validate application behavior
- Clean up test failover resources
- Perform and commit production failover

---

## 📋 Prerequisites

- Azure subscription with **Owner** or **Contributor** permissions
- Ability to create:
  - Resource groups
  - Virtual machines
  - Recovery Services Vault
  - Networking resources (VNet, Subnet, NIC, NSG rules)

---

# 🧪 Lab Steps

---

## 1. Initial Setup

### 1.1 Create a Resource Group

1. Create a resource group for the source VM.
   Suggested name:= **rg-fntech-asr-lab-eus-core**

---

### 1.2 Create a Virtual Machine

1. Deploy the source VM in **East US**.
   Suggested name: **vm-fntech-asr-lab-eus-app01**
2. Under Availability options, select:
➡️ No infrastructure redundancy required
(This ensures the VM is not pinned to an Availability Zone, which avoids ASR zone‑to‑zone limitations in East US.)
3. Complete the remaining VM configuration using default settings.
---

### 1.3 Create a Recovery Services Vault

1. Create a Recovery Services Vault in the source region.
   Suggested name: **rsv-fntech-asr-lab-eus-dr**

---

## 2. Enable Site Recovery Replication

### 2.1 Enable Disaster Recovery

1. In the VM blade, enable **Disaster Recovery**.
2. Set:
   - Source region: **East US**
   - Target region: **West US 2**
3. Do not change **Advanced settings**.
4. Note the cache storage account (auto-created).
5. Change source managed disk SKU to **Standard** if required.

> **Note:** The ASR cache storage account can be auto-generated by Azure and may not exactly match your preferred naming convention.

---

### 2.2 Review Target Resources to be Created

1. Confirm the following resources that will be created in the target location:
   - Target Resource Group: **rg-fntech-asr-lab-wus2-dr**
   - Target Virtual Network: **vnet-fntech-asr-lab-wus2-dr**
   - Target Subnet: **snet-fntech-asr-lab-wus2-dr**
   - Target NIC: **nic-fntech-asr-lab-wus2-dr01**
   - Target Managed Disk: **disk-fntech-asr-lab-wus2-dr01**
   - Cache Storage Account: **Auto-generated by Azure (record the generated name)**
   - Automation Account: **aa-fntech-asr-lab-eus-asr**
   - Replication Policy: **asrpol-fntech-asr-lab-eus-wus2**
   - (Later) Test Failover VM: **vm-fntech-asr-lab-wus2-test01**

---

### 2.3 Start Replication

1. Click **Start replication**.
2. Wait for replication to initialize (can take up to 30 minutes).

---

## 3. Application Validation Setup

### 3.1 Install IIS on the Source VM

1. Install the IIS role on the source VM.
2. Follow the setup steps in [1-build-base-vm.md](../VMSS/1-build-base-vm.md).

---

### 3.2 Add HTTP Inbound Rule

1. Add an inbound NSG rule:
   - Protocol: **TCP**
   - Port: **80**

---

### 3.3 Validate Website

1. Browse to the VM web endpoint.
2. Confirm the default Hello World/IIS page is reachable.

---

## 4. Replication Health Checks

### 4.1 Verify Target Resource Group

1. Confirm a new resource group is created in the target region.
2. Validate that target networking resources (including VNet) are present.

---

### 4.2 Verify Vault Resource Group Assets

1. In the Site Recovery vault resource group, verify:
   - Cache Storage Account
   - Automation Account
   - Recovery Services Vault

---

### 4.3 Verify Replicated Items and RPO

1. Check **Replicated items** and ensure status is **Healthy**.
2. Verify **RPO** value before running failover.

---

## 5. Test Failover

### 5.1 Update Application Content

1. Modify the IIS HTML content on the source VM.
2. Wait about 20 minutes to allow replication to catch up.

---

### 5.2 Run Test Failover

1. Start **Test failover**.
2. Select a recovery point.
3. Select the target test failover virtual network.
4. Launch test failover.

---

### 5.3 Validate Test VM

1. Verify test failover VM creation in target location.
2. Assign a Public IP to the test VM if needed.
3. Browse to the website and verify updated HTML content.

---

### 5.4 Clean Up Test Failover

1. Run **Cleanup test failover**.
2. Confirm:
   - Test failover completes successfully
   - Test Failover button becomes enabled again
   - Test VM is removed from target region

---

## 6. Perform Planned Failover

### 6.1 Initiate Failover

1. Decide failover type:
   - **Planned failover**: Use when source VM is available and can be shut down cleanly.
   - **Unplanned failover**: Use when source VM or source region is unavailable.
2. For planned failover, shut down the source VM/application gracefully.
3. Start **Failover**.
4. Change/select the desired recovery point.

---

### 6.2 Commit Failover

1. Click **Commit**.
2. Confirm target location is now primary.

---

### 6.3 Post-Failover Next Steps

1. Validate application health, network connectivity, and data state in the target region.
2. Enable **Re-protect** to start replication in the reverse direction.
3. Plan and execute **Failback** when the source environment is ready.
